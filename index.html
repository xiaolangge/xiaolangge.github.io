Maven多环境构建支持

Posted on 2021-05-02 In 构建工具

在我们的日常开发工作当中，一个系统通常会有多个环境：local本地开发环境，dev环境，qa环境，online环境等等。
很多配置或者某些服务只希望在相应的环境下才能被执行，本文接下来会讨论Maven的三大特性：属性，Profile和资源过滤。通过这三个特性可以帮助我们灵活的构建多环境系统。

  • POM属性:
  • 自定义属性:
  • settings属性
  • Java系统属性
  • 环境变量属性

Profile

pom.xml里面的profile可以看成是一个块，这个块里面可以自定义一堆东西，然后不同的块只能激活一个。

资源过滤

maven-resources-plugin插件默认只会将主资源文件和测试资源文件copy到target下相应的目录去，默认情况下是不会过滤资源文件。

三者结合起来

接下来我们可以将Maven属性，Profile和资源过滤三者结合起来，构建一个多环境的项目构建。具体步骤如下：

  • 将配置文件因环境有变化的部分以Maven属性的方式提取出来，如下：
  • 在pom.xml文件的profile里面配置这些属性值
  • 开启资源过滤
  • 激活Profile

激活Profile的方式

最后在讲下Maven Profile的激活方式，一共有六种激活方式。

  • 命令行激活, mvn clean package -Pdev-x,dev-y
  • settings文件显示激活, activeProfiles元素, 不常用.
  • 系统属性激活, mvn clean package -Dtest , 然后pom.xml里面的activation元素, 不常用.
  • 操作系统环境激活, 不常用.
  • 文件存在与否激活, 不常用.
  • 默认激活.
    记住命令行激活方式和默认激活方式，其他的可以忽略其存在。

查看当前激活的Profile

如果一个项目Profile管理很混乱，很多profiles，这时想要知道哪个profile被激活了怎么办? 使用下面的命令可以查看：

查看所有的Profiles: mvn help:all-profiles

profiles种类

  • 项目pom.xml的profile
  • 用户settings.xml的profile
  • 全局settings.xml的profile
    实际项目中，用的最多的肯定是项目下的pom.xml中的Profile这种形式。

Maven Archetype介绍

当工作中需要新建一个服务时，通常我们直接使用IDEA创建一个Maven项目或者利用Spring Initializer创建一个Spring项目。
它们本质都是Maven项目，都是使用Maven的一个插件maven-archetype-plugin，然后根据某个项目模板生成我们的服务骨架代码。
这个插件的版本信息可以在这里看到：https://mvnrepository.com/artifact/org.apache.maven.plugins/maven-archetype-plugin 最后，生成的项目骨架可能的样子：（采用的模板是：maven-archetype-quickstart）

解释下archetype，意思是原型。
maven-archetype-plugin：是maven的一个插件。
maven-archetype-quickstart/webapp/xxx：是项目模板。
那上面的命令是说，利用archetype-plugin插件根据archetype-quickstart模板生成我们自己项目的骨架代码。

使用mvn archetype生成项目骨架时要注意，等号左右两边不能有空格，如：-DarchetypeGroupId = org.apache.maven.archetypes这样是不行的，执行后会报错。当初我参照Maven实战书上的例子时就踩了这个坑。

还可以指定archetype插件的版本，如：mvn org.apache.maven.plugins:maven-archetype-plugin:3.2.0:generate这样来执行命令。主要是防止Maven2用户使用到老版本的Archetype。

下面列举了常见的两种模板类型：

  • maven-archetype-quickstart 这是普通的Java Standalone项目。
  • maven-archetype-webapp 这是Java Web项目
  • 更多模板: https://repo.maven.apache.org/maven2/archetype-catalog.xml

动手编写自己的模板

很多时候，我们项目组会有自己项目骨架，这个时候每次新建一个服务都要配置一遍很费时间和精力。编写一个自己的模板势在必行。
一个模板有三个部分：

  • 模板的pom.xml，模板也是maven项目也需要坐标定位。
  • 待生成项目的pom.xml
  • 待生成项目的目录结构的描述文件，它描述了生成项目有怎么样的结构等等。
    我已经实现了一个，可以基于它来改造：https://github.com/xiaolangge/myProject-archetype
    上面也有注释，很容易看得懂。使用我们自定义的模板生成新项目的骨架：

maven-archetype-plugin从哪里读取模板呢？

  • internal：内置的archetype catalog
  • local: ~/.m2/archetype-catalog.xml
  • remote: https://repo.maven.apache.org/maven2/archetype-catalog.xml
  • file://…: 指定本机地址
  • http://…: 以http协议指定远程的archetype-catalog.xml文件
    命令行指定archetypeCatalog参数，mvn archetype:genearte -DarchetypeCatalog=xxx; maven-archetype-plugin会有默认值，具体可以看下其文档，最后archetypeCatalog可以指定多个值。

基于本地仓库生成archetype catalog

默认方式:
mvn archetype:crawl
指定哪个本地仓库，生成的catalog的目录：
mvn archetype:crawl -Drepository=xxx/repository -Dcatalog=./archetype-catalog.xml
最后用户可以指定该catalog来生成项目骨架了。

当然自己也可以将模板推导中央仓库去，如果感兴趣可以去google一下。最后，分享一个spring-boot-archetype模板，它是一个多模块的spring-boot项目，包含了一些基本的controller/service等等，值得一试。地址： https://github.com/xiaolangge/spring-boot-archetype

时隔数年我为啥重返微信公众号

Posted on 2021-04-30 Edited on 2021-05-01 In 闲聊

大家好，我是小浪哥。目前我在百度这边做后台开发，同时也是某业务线的负责人，带领七八个小伙伴在一线战场奋斗着。今年已经是工作的第八个年头了，从双非渣本毕业经过摸爬滚打到现在的百度，前前后后混迹过几个大厂也在创业型公司待过，拥有很多个高并发系统的实战经验（支付系统，AppStore后台，广告系统DSP等等），这些系统有的是以Owner的身份从0到1建设起来的，有些是以核心开发者身份深度参与的，总之一路走来磕磕碰碰有很多感受和经验可以分享。

言归正传吧，几年之前我也折腾过微信公众号，后来不了了之注销了。那为啥现在要重返微信公众号呢？原因有三个吧。

第一：有了更多的实战经验和体会想分享。尤其在百度这边，从刚开始按指令做事到现在要以多个视角/身份看问题并要带领其他同学一起前进，有了更多的责任当然也有了更多的体会。

第二：梳理自己的技术体系和总结职场经验和体会。目前在百度这边可谓相当的忙，每天有开不完的会，永远都做不完的需求等等。我觉得利用深夜的时间找个平台梳理下技术和职场那些事也蛮好的。

第三：做一个zui好的技术微信公众号。（好吧，我内心的确是这么想的）为啥这么说呢，平时我上班路上也会看一些技术方面的微信公众号，最近也取关了很多公众号，它们老是推一些极客时间课程相关的，割一把韭菜。我不是说这些课程不好，我自己手机里面也有极客时间app，总之老是推这些我很反感。

最后聊聊我这个微信公众号的聚焦吧，我想把它聚焦在两个部分，一是分享一些职场的东西称闲聊吧，二是以Topic形式来分享技术，很多时候我们接触的知识都太碎片化了，为此我想以topic形式将一线开发的技术串起来呈现给读者。
